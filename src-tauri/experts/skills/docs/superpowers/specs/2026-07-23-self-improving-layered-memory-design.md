# Self-Improving 分层记忆设计

日期：2026-07-23

状态：已批准

## 背景

现有 `self-improving` 只在用户明确纠正、表达长期偏好或事实、要求记住信息，
以及 Agent 需要记录执行反思时触发。它已经使用 iyw-claw 的权威用户记忆工具，
但触发描述偏保守，且没有清晰表达候选记忆、确认记忆、用户画像和交互准则之间的
晋升关系。

本次调整需要提高 skill 对长期有用用户信息的自动触发概率，同时防止误记、过度
收集、敏感信息持久化和另建并行记忆库。

## 目标

1. 用户自然表达可复用信息时，主动识别并路由到合适的记忆层。
2. 默认先生成可审核候选，仅在用户明确要求记住或确认候选后写入确认记忆。
3. 明确当前会话、候选记忆、确认记忆、用户画像和交互准则的边界。
4. 继续使用 iyw-claw 的 host memory tools，不直接编辑权威记忆文件。
5. 保持 Agent 操作反思与用户记忆彻底隔离。

## 非目标

- 不从沉默、语气、情绪、点击、访问记录或其他行为推断用户特征。
- 不主动盘问用户以完善画像。
- 不保存凭据、敏感属性、第三方个人信息或一次性任务内容。
- 不在 `~/.iyw-claw/self-improving/` 下创建第二套用户记忆数据库。
- 不通过 skill 声称保证每轮对话都执行记忆扫描。
- 不在本仓库实现 iyw-claw 宿主的全局 Memory Observer。

## 方案选择

采用五层记忆漏斗。与置信度自动晋升或按领域拆分多个画像相比，该方案可以复用
现有 host memory tools，并把默认动作限制为可审核提议。

| 层级 | 内容 | 持久化与权限 |
| --- | --- | --- |
| M0 会话上下文 | 当前任务、临时要求、短期状态 | 只存在于当前会话，不持久化 |
| M1 候选记忆 | 用户明确表达的可复用偏好、事实或纠正 | 调用 `propose_user_memory`，等待用户审核 |
| M2 确认记忆 | 用户明确要求记住，或已经批准的候选 | 调用 `append_user_memory` 或由宿主完成候选批准 |
| M3 用户画像 | 从多条确认记忆归纳的稳定用户特征 | 仅由 iyw-claw 管理 `user-profile.md` |
| M4 交互准则 | 高稳定度的沟通方式、价值取向和长期边界 | 仅由 iyw-claw 管理 `user-soul.md` |

Agent 自身的执行经验属于独立的 Operational Reflection 层。它可以写入
`~/.iyw-claw/self-improving/reflections.md`，但不得包含、复制或归纳用户信息。

## 自动触发策略

更新 `SKILL.md` frontmatter 的 description，使 skill 在以下信号出现时主动使用：

- 用户要求记住、保存或以后沿用某项信息。
- 用户以第一人称明确表达跨任务可复用的偏好、事实、长期目标或固定约束。
- 用户明确纠正 Agent 的长期做法。
- 用户在当前可见上下文中再次表达已经出现过的长期偏好或事实。
- 重大工作、失败或纠正后需要记录不包含用户信息的操作反思。

Description 必须同时写明主动使用场景和隐私边界。自动触发是模型基于 skill
metadata 的 best-effort 行为。若产品要求每轮消息都执行记忆分类，必须由 iyw-claw
宿主增加全局 Memory Observer；skill 本身不能提供强保证。

## 分类与路由

每次 skill 被触发时，按以下顺序分类：

1. 过滤禁止保存的敏感信息、秘密、第三方信息和行为推断。
2. 判断信息是否只对当前任务有效。一次性指令保留在 M0。
3. 判断用户是否明确要求记住。
4. 明确要求记住的非敏感长期信息进入 M2。
5. 其他明确且可复用的信息进入 M1，等待用户审核。
6. 没有可复用信息时不调用记忆工具，不制造提示或打断当前任务。

信号路由如下：

| 用户信号 | 路由 |
| --- | --- |
| “记住我偏好简洁回答” | `append_user_memory`，写入确认偏好 |
| “我通常偏好简洁回答” | `propose_user_memory`，signal 为 `preference` |
| “以后不要自动改根配置” | 根据语义提出 `correction` 或 `preference` 候选 |
| “这次使用中文” | M0，不持久化 |
| 重复表达同一长期偏好 | 去重；必要时保持单个候选，不自动晋升 |
| 与旧记忆冲突的新说法 | 当前任务采用新说法，并提出更新候选，不静默覆盖 |
| 令牌、密码、医疗信息或第三方信息 | 拒绝持久化，不转写或摘要保存 |

## 晋升、去重与冲突

- M1 只能在用户批准后晋升为 M2。
- 用户明确说“记住”时可以直接写入 M2，无需先创建 M1。
- 重复出现表示值得再次确认，不等同于用户授权自动保存。
- 依赖 host memory tools 的去重、审核、备份和用户控制能力。
- 当前用户和项目指令始终高于旧记忆。
- 新旧信息冲突时提交新的候选更新；不得直接编辑或重写记忆文件。
- 工具不可用时不落盘，并明确说明未改变长期记忆。

## 隐私与控制

以下内容永不进入 M1 至 M4：

- 密码、令牌、密钥、支付和金融数据。
- 医疗、健康、生物识别和精确位置规律。
- 推断出的敏感属性、情绪或人格结论。
- 第三方个人信息和访问行为。
- 仓库事实、临时进度、一次性任务参数和 Agent 私有推理。

用户可以通过 iyw-claw 的 User Memory 设置或受支持的 host operations 查看、批准、
拒绝、删除和导出记忆。Skill 不直接操作 `user-memory.md`、`user-profile.md` 或
`user-soul.md`。

## 文件变更范围

- `self-improving/SKILL.md`：扩展触发 description，加入分层模型和低噪声路由。
- `self-improving/learning.md`：定义候选识别、分类和晋升规则。
- `self-improving/memory.md`：记录五层模型与宿主所有权。
- `self-improving/operations.md`：补充候选、确认、冲突和工具不可用处理。
- `self-improving/boundaries.md`：补充禁止主动盘问、行为推断和自动晋升。

不修改 `_meta.json`，不新增记忆存储脚本，不改变 heartbeat 对用户记忆的隔离规则。

## 验证与验收

实现完成后执行：

1. 使用 `quick_validate.py` 校验 skill frontmatter 和目录结构。
2. 检查 `SKILL.md` description 包含所有主动触发信号和隐私限制。
3. 检查五个目标文档对 M0 至 M4、工具名称和晋升条件的描述一致。
4. 检查 heartbeat 文档仍禁止读取、整理或删除用户记忆。
5. 用以下场景做定向验证：

   - 明确“记住”调用 append 路由。
   - 自然表达长期偏好调用 propose 路由。
   - 一次性任务要求不持久化。
   - 重复偏好不自动晋升。
   - 冲突偏好提出更新候选。
   - 敏感信息和第三方信息不持久化。
   - host tool 不可用时不直接写文件。

验收标准：skill 能更积极地捕获用户自然表达的长期信息，但所有非明确记住请求都
默认进入可审核候选层；不新增并行数据库，不弱化现有隐私边界。
